Critical Decisions That Shaped Development

Decision 1: Philosophy First
You spent 4 hours clarifying product philosophy before touching code. This saved dozens of revision cycles later. The philosophy document became the single source of truth for every design decision.
Decision 2: Local-First Architecture
Instead of building a cloud-based system, you chose local-first JSON storage. This reduced complexity by 80%, eliminated server costs, and made the product instantly private-by-default.
Decision 3: Pivot to PWA
After building React Native, you quickly built a PWA version for immediate web deployment. The pivot cost 24 extra hours but gave you a deployed product immediately (vs. 3-7 days for app store reviews).
Decision 4: Aggressive QA Testing
You tested ruthlessly on real devices. This found 19 bugs that simulators missed. Every bug fixed before users saw them = confidence in the product.

⚡ Token-Saving Prompt Patterns

  1. Reference External Documents
    Instead of: "Here's my 2000-word philosophy doc, can you read it..."
    Use: "See the philosophy doc at /path. Now build the UI following principle #3."
    Saves 1000+ tokens
  2. Provide Code Diffs, Not Full Files
    Instead of pasting 500 lines of code asking for a fix
    Use: "In the Home.tsx file, lines 45-60 have a bug. The issue is X. Fix it."
    Saves 400+ tokens
  3. Use Structured Briefs
    Instead of: "The app has 5 screens and..."
    Use formatted sections with headers and lists
    Saves 300+ tokens
  4. Clear Acceptance Criteria
    "Create the button that: [1] Has cyan background [2] Shows loading state [3] Calls API on click"
    vs. "Make a nice button for submitting"
    Saves 200+ tokens + reduces revisions
  5. Ask for One Thing Per Message
    Instead of: "Create app, fix bugs, optimize performance, add tests"
    Use: "Create app." (then next message) "Now fix bugs in Home.tsx"
    Saves 500+ tokens

📚 8 Core Lessons from Spend-Na

These lessons will apply to every product you build with AI in the future.

Lesson 1: Philosophy First, Code Second ⭐ Most Important

What You Did Right: You spent 4 hours on philosophy before writing code.

Why This Mattered:
Clarity on values shaped every design decision. When a bug appeared (like "should the keyboard stay open?"), the answer was instant: "Does it serve user awareness? Yes → keep it open." No philosophy = endless debates.

Apply This To Future Products:

  • Write a 1-2 page philosophy doc BEFORE asking me to code
  • Include: Problem you're solving, who it's for, core values, what you're NOT building
  • Share it with me for feedback before implementation

Lesson 2: Separate Concerns Across Sessions

What You Did Right: Each day had a clear theme (Day 1: Philosophy, Day 2: Architecture, Day 3: Code, Day 4: Testing)

Why This Mattered:
Session context is limited. By separating concerns, each session was laser-focused. Day 3 didn't need to rehash Day 1's philosophy—it was the contract.

Apply This To Future Products:

  • Session 1: Problem definition & philosophy
  • Session 2: Architecture & tech selection
  • Session 3: Code implementation
  • Session 4: Testing & refinement

Lesson 3: Conversation Over Specification

What You Did Right: You engaged in thinking partnership, not spec handoff.

Why This Mattered:
Detailed specs miss edge cases. Conversation revealed assumptions. ("Should the app work offline?" → thinking partnership → "Yes, local-first is the whole idea" → changes architecture)

Apply This To Future Products:

  • Don't write a 50-page spec and disappear
  • Bring your thinking, ask questions, challenge my assumptions
  • The best ideas came from "what if..." conversations

Lesson 4: Test On Real Devices Early & Often

What You Did Right: You tested on real iPhones, not just simulators.

Why This Mattered:
The iOS SafeAreaProvider bug only showed on real devices. Simulator said "works fine." This taught you: real devices reveal 80% of bugs. Simulator catches 20%.

Apply This To Future Products:

  • Always test on real hardware before calling it done
  • Test on the oldest devices you support (found accessibility issues this way)
  • Real device testing = best ROI for your time

Lesson 5: Document Decisions, Not Just Code

What You Did Right: You wrote the SpendnaLearningGuide.md explaining WHY you chose React Native.

Why This Mattered:
Six months from now, you won't remember why you chose local-first. Documentation is your future self saying "here's why."

Apply This To Future Products:

  • For each major decision, write 2-3 sentences in a "decisions.md" file
  • Include: Alternative considered, why you chose this one, trade-offs
  • This becomes your knowledge base and helps onboard future collaborators

Lesson 6: Token Economy Matters (Attention Cost)

What You Did Right: You moved from vague prompts → structured briefs → specific requests.

Why This Mattered:
Each vague prompt wastes tokens on clarification. Structured briefs = fewer tokens + better output. You went from 40K tokens/day to 25K/day by day 3 (same output, better prompts).

Apply This To Future Products:

  • Template every brief with: Context → Problem → Solution → Constraints → Question
  • Include acceptance criteria (what "done" looks like)
  • One thing per prompt (except related follow-ups)

Lesson 7: Constraints Are Features

What You Did Right: "Local-first, zero server cost, works offline" became your biggest differentiator.

Why This Mattered:
Constraints forced creative solutions. No server → forced simplicity in data model. Forced simplicity → easier to maintain, easier to pivot.

Apply This To Future Products:

  • Embrace constraints. They're not limiting, they're clarifying
  • Instead of "make it scalable," say "must work with 100K concurrent users"
  • Constraints = better product focus

Lesson 8: Share Blank App, Not Personal Data

What You Did Right: Users can share the app without exposing their financial data.

Why This Mattered:
This is a product design philosophy win, not just a technical one. Privacy-first = trustworthy = people actually use it.

Apply This To Future Products:

  • Think about sharing at design time, not after
  • Privacy constraints often lead to better UX

✨ Advanced Prompting Tricks for Next Time

These techniques will make you 3-5x faster with AI on the next project.

1The Philosophy Contract

Write a 1-2 page philosophy doc BEFORE asking for code. Include: problem, who it's for, core values, trade-offs you've decided on. Share it with me for feedback.

Token Savings: 10K+ (fewer revisions)

2Specific Not Vague

Instead of "create a login screen," say "create a login screen with: [1] email/password fields [2] 'remember me' checkbox [3] loading state during API call [4] error messages in red, centered."

Token Savings: 3K+ (fewer back-and-forths)

3Use Diffs, Not Whole Files

Instead of pasting 500 lines of code, paste just the broken section: "In Home.tsx line 45-60, the button click doesn't work. Fix the onClick handler."

Token Savings: 5K+ (fewer tokens per request)

4One Thing Per Prompt

Don't ask for 5 things at once. Ask for 1 thing, get it done, then ask for the next. This keeps context clean and reduces mistakes.

Token Savings: 2K+ (fewer context misses)

5Provide Acceptance Criteria

End every request with "Done when: [1] ... [2] ... [3] ..." This removes ambiguity and speeds up iteration.

Token Savings: 4K+ (no revisions for misunderstanding)

6Reference External Docs

If your spec is long, save it and reference it: "See the architecture doc at /path/. Now implement service 1 following principle 3." Don't paste it every time.

Token Savings: 8K+ (don't repeat context)

7Constraints First, Features Second

Instead of "build a fast app," say "must load in < 2 seconds on 4G, work offline, use < 5MB data/month." Constraints → better decisions.

Token Savings: 6K+ (don't rebuild for hidden constraints)

8Test Plan Before Code

Write test cases BEFORE asking for implementation. "When user clicks button without filling email, show 'Email required' error." I can build to spec instead of guessing.

Token Savings: 7K+ (fewer bug fix iterations)

9Decision Documentation

Keep a decisions.md file. Every time you choose A over B, document it: "Why local-first: [benefits], alternatives: [what we rejected], trade-offs: [what we lose]"

Token Savings: 3K+ (no rehashing decisions)

10Phase Gate Reviews

At the end of each phase, agree on what "done" means. "Phase 1 complete when: [1] philosophy doc written [2] you've read it [3] you've given feedback." Don't move forward until aligned.

Token Savings: 5K+ (avoid rework)

⚠️ Anti-Patterns to Avoid Next Time

❌ Don't: "Here's my code, it's broken, fix it."
✅ Do: "In Home.tsx line 45, the button click handler doesn't fire. I think it's because useState isn't imported. Check and fix."
You gave me context, narrowed the problem, even hypothesized the cause. Saves me time and tokens.
❌ Don't: Ask the same architecture question 3 times with different wording.
✅ Do: Ask once, get an answer, move forward. If you want to revisit, say "I'm reconsidering. Here's why..." (takes 30 seconds, saves confusion).
Flip-flopping wastes tokens. Make a decision and own it.
❌ Don't: Paste entire 300-line files to ask about 10 lines.
✅ Do: Extract the relevant section and comment it.
I have to read all 300 lines to find the bug. Just give me the 10.

🏗️ Your Reusable Framework

Use this exact structure for every product you build with AI. It worked for Spend-Na, it will work for the next 10 products.

The 5-Phase AI-Human Product Development Framework

📋 Phase 1: Philosophy (4-8 hours)

Goal: Define the product at a philosophical level, not technical.

Outputs:
  • 1-2 page philosophy document
  • Problem statement (the itch you're scratching)
  • User persona (who this is for)
  • Core values (what guides decisions)
  • Trade-offs you've decided on (e.g., "privacy over convenience")
  • What you're NOT building
How to Brief Claude:
"Here's my philosophy doc. Does it make sense? What am I missing? Any blind spots?" (Don't ask for code yet)

🏛️ Phase 2: Architecture (6-12 hours)

Goal: Choose technology, design system, and high-level architecture.

Inputs from Phase 1: Philosophy doc

Outputs:
  • Technology stack (frontend, backend, database, deployment)
  • System architecture diagram
  • Data model (key entities and relationships)
  • API design (if applicable)
  • Deployment strategy
  • Constraints document (performance, privacy, scale, cost)
How to Brief Claude:
"I need a tech stack for [product]. Constraints: [list]. Background: [your expertise]. Recommend and justify each choice."

💻 Phase 3: Development (24-48 hours)

Goal: Build the MVP. Split into focused tasks.

Inputs from Phase 2: Architecture, constraints, tech stack

Outputs:
  • Working MVP with core features
  • Code repository (GitHub, GitLab, etc.)
  • Deployment (even if just to localhost)
  • README with setup instructions
How to Brief Claude:
"Build screen 1 (Home). Done when: [acceptance criteria]. Philosophy reminder: [key principle]."

Pro Tip: Don't ask for everything at once. Screen 1, then screen 2, then integration. This keeps context clean.

🧪 Phase 4: Testing & QA (12-24 hours)

Goal: Find bugs, fix them, document issues.

Inputs from Phase 3: Working MVP

Outputs:
  • QA report (what you tested, what broke, severity)
  • All critical bugs fixed
  • Test plan (for future regression testing)
  • Known limitations document
How to Brief Claude:
"I found a bug: [exact steps to reproduce]. Here's a screenshot. Can you fix it?"

Pro Tip: Test on REAL devices. Don't ask me to fix bugs you haven't verified on real hardware.

🚀 Phase 5: Deployment (8-16 hours)

Goal: Get the product in front of users.

Inputs from Phase 4: Tested, bug-free MVP

Outputs:
  • Live product (web, app store, or both)
  • Deployment guide (how to update in future)
  • User onboarding (if applicable)
  • Release notes
How to Brief Claude:
"Deploy the app to Netlify. Include deployment instructions so I can do it myself next time."

📝 Checklist for Each Phase

Before starting a phase, ask:

  • ☑️ Do I have all inputs from the previous phase?
  • ☑️ Have I shared the philosophy/requirements with Claude?
  • ☑️ Have I defined what "done" means for this phase?
  • ☑️ Do I have acceptance criteria (not vague)?
  • ☑️ Is Claude clear on constraints?

During a phase:

  • ☑️ One thing per prompt (or related follow-ups)
  • ☑️ Use diffs, not whole files
  • ☑️ Document decisions as you go
  • ☑️ Test on real devices (Phase 3+)
  • ☑️ Reference the philosophy doc when you disagree

At end of phase:

  • ☑️ Do all outputs match the definition of "done"?
  • ☑️ Have I documented what was built and why?
  • ☑️ Are there any technical debts or known issues?
  • ☑️ Is the next phase's input clear?

⚠️ Common Mistakes to Avoid

Mistake 1: Skipping Phase 1
You can't skip philosophy. You'll build the wrong thing. Spend the time upfront.
Mistake 2: Changing Your Mind Mid-Phase
If you want to pivot, finish the phase, document the decision, then start a new phase. Don't thrash mid-phase.
Mistake 3: Testing Only in Simulator
Real devices are 80% of your bugs. Don't launch without real device testing.
Mistake 4: Vague Prompts
"Build a feature" will waste more tokens than "Build X with constraints Y, Z. Done when..."
Mistake 5: Not Using Diffs
Pasting entire files wastes tokens. Always extract the 10 relevant lines.
